First question: Local UI — how does it live on your machine?

This is the most foundational frontend decision. It affects packaging, hotkeys, window management, tray icons, and how updates work.

A

Electron desktop app

A native desktop window — installed like any app. Can register global hotkeys, live in the system tray, show native notifications, manage multiple windows. The Vite frontend compiles to a bundle loaded inside Electron. No browser needed.

Pros

  • True desktop app feel — tray, hotkeys, native popups
  • No port conflicts or browser juggling
  • Can control window position/size programmatically
  • IPC channel to FastAPI backend

Cons

  • Heavier packaging (~150MB baseline)
  • Separate local vs remote build targets
  • Auto-update infrastructure needed
B

Browser-served web app (localhost)

Vite app served from a local webserver (Nginx or Node). Open http://localhost:PORT in any browser. Remote version is the same codebase deployed to a server. The "local" and "remote" builds are identical — just different host targets.

Pros

  • Single codebase — local and remote are identical
  • No packaging — just a running service
  • DevTools always available
  • Easier to update (just rebuild/restart)

Cons

  • Global hotkeys need a separate daemon (Python-side)
  • Window management via browser APIs only
  • Popups are browser popups, not native windows
C

Tauri desktop app

Like Electron but using Rust + the OS's native webview instead of Chromium. Much smaller binary (~5MB). Same Vite frontend compiles directly. Still a true desktop app with tray, hotkeys, and native windows — just lighter.

Pros

  • Tiny binary — no Chromium bundled
  • Same native desktop capabilities as Electron
  • Vite frontend works unchanged
  • Strong security model

Cons

  • Requires Rust toolchain for builds
  • Less ecosystem maturity than Electron
  • WebView rendering can differ across OS versions
